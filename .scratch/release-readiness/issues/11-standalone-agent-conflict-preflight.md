## Agent Brief

**Category:** bug
**Summary:** Fail closed before installation when standalone custom-agent files
claim a router-managed role name.

**Current behavior:**
Installation planning validates managed role names only in the primary Codex
configuration document. Standalone custom-agent definitions under the explicit
Codex home's agent directory are ignored. A plan can therefore report no
conflicts while Codex has another active definition for the same managed role,
including definitions that pin model, effort, or developer instructions contrary
to the router's role/compute separation invariant.

**Desired behavior:**
Planning, installation, and health reporting must treat every active standalone
custom-agent definition as part of the role-conflict surface. Until the project
explicitly supports adopting that configuration form, any standalone agent whose
declared `name` matches a managed role must produce a clear fail-closed conflict.
The installer must never rewrite, remove, adopt, snapshot, or claim ownership of
those user-owned files.

**Key interfaces:**
- The installation plan's managed-role conflict result
- The locked install path that repeats preflight checks before the first write
- Installation status details for configuration added or changed after install
- Managed role contracts as the single source of managed role names
- Rollback ownership, which remains limited to installer-created role blocks,
  hook groups, and transaction state

**Acceptance criteria:**
- [ ] A standalone TOML definition whose declared `name` equals any managed role
      produces an explicit plan conflict, regardless of its filename.
- [ ] The install operation repeats the standalone-role check and writes no user
      configuration or receipt when a collision exists or appears after plan.
- [ ] Installation status becomes unhealthy with an actionable detail when a
      same-name standalone definition appears after a healthy install.
- [ ] Standalone definitions for unmanaged names remain untouched and do not
      conflict solely because they exist.
- [ ] Unreadable, unsafe, or invalid standalone configuration fails closed when
      the installer cannot prove that it does not claim a managed role.
- [ ] Rollback never modifies or removes files in the standalone agent directory.
- [ ] Tests cover a filename that differs from the declared agent name, a managed
      collision, an unmanaged definition, invalid input, install-time recheck,
      status drift, and rollback preservation using only temporary Codex homes.
- [ ] Existing plan/install/status/rollback behavior for `config.toml` and
      `hooks.json` remains unchanged when no standalone collision exists.

**Out of scope:**
- Migrating or adopting standalone custom-agent definitions
- Renaming the router's managed roles
- Selecting model, effort, sandbox, or developer instructions from agent files
- Declaring Codex 0.144.4 compatible; that is tracked by the separate release gate
